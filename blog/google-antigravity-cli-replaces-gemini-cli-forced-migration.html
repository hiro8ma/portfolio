Gemini CLI 終了 — Antigravity CLI への強制移行が CI/CD を壊す日

googlegeminiclidevtoolsci-cd

概要

2026年6月18日、Google が Gemini CLI の consumer 向け提供を停止した。後継は Go 製の Antigravity CLI だ。デスクトップ版 Antigravity 2.0 と同じ agent harness を共有する、別バイナリとして提供される。

問題は移行の強制力にある。Google は「launch 時点で 1:1 の feature parity はない」と明言した。Gemini CLI を組み込んだスクリプトや CI/CD パイプラインは、6月18日を境に動かなくなる。

これは単なるツール更新ではない。ベンダーの CLI に依存した自動化が、ベンダーの都合で一斉に壊れた事例だ。

詳細

誰が止まり、誰が残るか

停止対象は consumer 層に限られる。enterprise は猶予される。

区分対象6月18日以降
consumerGoogle AI Pro / Ultra subscriberGemini CLI 停止
consumerGemini Code Assist for individuals（無料枠）停止
consumerCode Assist for GitHub新規インストール不可
enterpriseCode Assist Standard / Enterprise継続利用可
enterprise有償 Gemini API キー経由継続利用可

個人開発者と小規模チームが最も影響を受ける。enterprise 契約があれば当面は逃げ切れるが、移行の先送りに過ぎない。

Gemini CLI と Antigravity CLI の違い

新 CLI は刷新されており、互換ではない。

観点Gemini CLIAntigravity CLI
実装言語Node.js 系Go（単一バイナリ）
実行モデル同期中心非同期 multi-agent
アーキテクチャ単独ツールAntigravity 2.0 とharness共有
機能パリティ—launch 時点で非対応あり
拡張Extensionsplugins（名称・仕組みが変更）

Agent Skills・Hooks・Subagents・Extensions は plugins として引き継がれる。だが「引き継がれる」ことと「同じコマンド・同じ引数で動く」ことは別だ。スクリプトレベルでは破壊的変更になる。

なぜ CI/CD が壊れるのか

CLI をパイプラインに組み込んでいる場合、バイナリ名・サブコマンド・認証フローの変更が直撃する。

# Before: Gemini CLI を前提にした CI ステップ
- name: AI code review
  run: |
    gemini auth login --key "$GEMINI_API_KEY"
    gemini review --diff origin/main --format json > review.json

# After: Antigravity CLI へ。コマンド体系が変わり、そのままでは動かない
- name: AI code review
  run: |
    antigravity auth --api-key "$GEMINI_API_KEY"   # 認証フローが変更
    antigravity agent run review --base origin/main # サブコマンド再設計

パイプラインを止めないには、6月18日より前に CLI 参照を全て洗い出し、新コマンドへ書き換え、staging で検証する必要があった。準備ができていないチームは、ビルドが赤くなって初めて気づく。

背景: コーディングツールの寡占化

この強制移行は、単独の出来事ではない。AI コーディングツールが巨大プラットフォームへ吸収される流れの一部だ。

出来事時期内容
Gemini CLI → Antigravity CLI6月18日Google が自社 agent 基盤へ統合
SpaceX が Cursor を買収6月16日$60B の全株式取引
独立系ツールの後退2026年主要コーディングツールが軒並み大手傘下に

独立したコーディングツールが大手の agent エコシステムへ統合されるほど、開発者は「ツールの仕様変更を自分で制御できない」状況に置かれる。Antigravity CLI への移行は、その構造を体感させる最初の請求書だ。

日本への示唆

日本企業にとって、これは2つの教訓を突きつける。

第一に、ベンダー CLI への直接依存はリスクだ。AI コーディングツールを業務フローや CI/CD に組み込む企業が増えた。だが CLI のコマンド体系はベンダーの裁量で変わる。今回のように猶予が短いと、移行が技術的負債ではなく即時の障害になる。

第二に、抽象化レイヤの欠如が露呈する。CLI を直接叩くのではなく、社内ラッパーや adapter 層を挟んでいれば、影響範囲を1か所に閉じ込められた。日本の多くの現場は「動いているから」と直接呼び出しを放置しがちだ。

観点直接依存型抽象化レイヤ型
移行時の修正箇所パイプライン全箇所adapter 1か所
ベンダー変更への耐性低い高い
初期コスト低い中程度
障害発生時の影響全 CI が停止局所化できる

対策は明快だ。AI CLI を業務に組み込むなら、コマンドを直接書かず、薄い wrapper を1枚挟む。ベンダーの仕様変更を、その1枚で吸収する設計にする。これは「最強モデルを選ぶ」議論より、はるかに地味で、はるかに効く。

まとめ

  • 2026年6月18日、Google が Gemini CLI の consumer 向け提供を停止し、Go 製 Antigravity CLI へ移行
  • enterprise（Code Assist Standard/Enterprise・有償 API キー）は継続利用可。個人・小規模が直撃
  • 新 CLI は非同期 multi-agent で刷新。launch 時点で 1:1 の機能パリティはなく、コマンド体系が破壊的に変わる
  • CLI を組み込んだ CI/CD は、バイナリ名・サブコマンド・認証フローの変更で停止する
  • 背景はコーディングツールの寡占化（SpaceX の Cursor 買収など）。仕様変更を開発者が制御できない構造が進む
  • 日本への教訓: ベンダー CLI を直接叩かず、adapter 層を1枚挟んで仕様変更を局所化する

便利な CLI ほど、深く業務に食い込む。そして食い込んだ分だけ、ベンダーの都合で壊れたときの被害は大きい。
