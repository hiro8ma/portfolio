## 1. 対象読者

| 属性 | 詳細 |
|------|------|
| **職種** | エンジニア、テック投資家、業界ウォッチャー |
| **技術レベル** | 中〜上級。SaaS, CI/CD, LLM 等の基本概念は説明不要 |
| **言語** | 日本語話者。英語の技術用語には慣れている |

## 2. ネタの選び方

### 対象トピック

| カテゴリ | 例 |
|----------|-----|
| **AI & 雇用** | レイオフ、自動化による職種変化、労働市場調査 |
| **ツール & リリース** | Claude Code, Codex, Go, Docker 等の新機能・アップデート |
| **ビジネス & 市場** | SaaS市場、AI投資、企業戦略の変化 |
| **開発プラクティス** | コードレビュー、Spec-Driven、エージェント活用 |
| **政策 & 倫理** | AI規制、政府対応、企業の政治的スタンス |

### 選定基準

| 条件 | 判断 |
|------|------|
| Xの1ツイートに収まらない深さがある | **書く** |
| 海外記事で日本への示唆を加える余地がある | **書く** |
| 数字・データがあり、テーブルで整理できる | **書く** |
| 既に content/posts/ に類似記事がある | **スキップ** |
| 単なるマイナーアップデート | **スキップ**（X投稿で十分） |

## 3. フロントマター

```yaml
---
title: "メインタイトル — サブタイトル"
date: "YYYY-MM-DD"
description: "1〜2文の要約。検索結果やOGPに表示される"
tags: ["タグ1", "タグ2", "タグ3"]
---
```

### タイトルの付け方

| パターン | 例 |
|----------|-----|
| **プロダクト + 変化** | `Claude Code Voice Mode 登場 — タイピングの3.7倍速でAIコーディング` |
| **現象 + インパクト** | `「AI リストラ元年」— Block 40%削減、みずほ配置転換が示す二極化` |
| **問い + 結論** | `コードレビューは死んだのか — Spec-Driven Development への転換` |

### タグの慣習

- 英語小文字、ハイフン区切り（`claude-code`, `ai-agents`）
- 企業名は小文字（`anthropic`, `google`, `openai`）
- 3〜5個が目安

## 4. 記事構成

### 基本テンプレート

```
## 概要（何が起きたか）
[日付・企業名・数字で始める。1〜2段落]

## 詳細（どういう内容か）
### サブトピック1
[テーブルやコード例で整理]
### サブトピック2
[比較、Before/After]

## 日本への示唆（なぜ自分ごとか）
[日本市場・日本のエンジニアへの具体的な影響]

## まとめ
[箇条書きまたは短い段落。3〜5個のキーテイクアウェイ]

## 参考リンク
- [ソース名](URL)
```

### 構成パターンの使い分け

| 記事タイプ | 構成 |
|-----------|------|
| **事件・発表系** | 概要（何が起きた）→ 背景（なぜ起きた）→ 影響（誰に効く）→ 日本への示唆 → まとめ |
| **技術リリース系** | 概要 → 主要機能（テーブル）→ 使い方（コード例）→ 所感 → まとめ |
| **トレンド分析系** | 問題提起 → データ/証拠（テーブル）→ 構造分析 → 日本への示唆 → まとめ |

## 5. 文体ルール

### 基本姿勢

| ルール | 説明 |
|--------|------|
| **断定調** | 「〜と思います」「〜かもしれません」は使わない |
| **事実ベース** | 数字・ソース・引用で裏付ける |
| **インサイダー視点** | 業界の内部者として書く。外から眺めるトーンは避ける |
| **適度な緊張感** | 破壊的な変化には urgency を出す。ただし煽りすぎない |

### 段落

- **1〜3文** が基本。長い散文ブロックは避ける
- 重要なポイントは **1文で1段落** にして目立たせる
- セクション間は `---` ではなく見出し（`##`, `###`）で区切る

### 見出し

- `##` でメジャーセクション（概要、詳細、日本への示唆、まとめ）
- `###` でサブトピック
- **見出しだけで記事の要旨が分かる** ようにする
- 例: `### /loop コマンド — 定期プロンプト実行`（機能名 + 一言説明）

## 6. テーブルの使い方

### 頻度

- **1記事に3〜5個** が目安
- テーブルはこのブログの最も特徴的な要素

### 用途別パターン

#### 比較テーブル

```markdown
| 項目 | Block（米国） | みずほ（日本） |
|------|-------------|--------------|
| 削減規模 | 全従業員の約40% | 配置転換（新規採用抑制） |
| 速度 | 即時解雇 | 10年計画 |
| 株価反応 | +24% | 横ばい |
```

#### データテーブル

```markdown
| 職種 | AI代替リスク | 根拠 |
|------|------------|------|
| データ入力 | 高 | 定型作業の自動化 |
| ソフトウェア開発 | 中 | コード生成は進むが設計判断は残る |
| 経営戦略 | 低 | 不確実性下の意思決定 |
```

#### Before/After テーブル

```markdown
| 観点 | Before | After |
|------|--------|-------|
| レビュー | 人間が全行チェック | AI が初期スクリーニング |
| 所要時間 | 2〜3時間 | 20分 |
```

#### 機能一覧テーブル

```markdown
| 機能 | 説明 | 対応状況 |
|------|------|---------|
| Voice Mode | 音声でコーディング指示 | ✅ GA |
| Agent Teams | 複数エージェント協調 | ✅ GA |
| Code Review | マルチエージェントレビュー | 🔬 Preview |
```

## 7. コードブロック

### 使い方

- 技術リリース・ツール紹介記事では **必須**
- 言語指定付き（```bash, ```go, ```json 等）
- **5〜20行** が目安。長すぎるコードは抜粋する
- Before/After の比較で使うと効果的

### 例

```bash
# Voice Mode の起動
claude --voice

# 定期実行
/loop 5m check the deploy status
```

```go
// Green Tea GC の恩恵を受けるコード
func process(data []byte) {
    result := transform(data)
    // GC pause が 50% 削減
    send(result)
}
```

## 8. 引用（ブロッククォート）

### 使うとき

- **経営者・研究者の直接発言** のみ
- 記事の論旨を支える重要な引用に限定

### フォーマット

```markdown
> 「インテリジェンスツールは、会社を構築し運営することの意味を変えた」
> — Darren Mowry, Block CTO
```

### 使わないとき

- 自分の意見や要約（地の文で書く）
- 長い文章の丸ごと転載

## 9. 日本語・英語の混ぜ方

### 比率

- **日本語 70% / 英語 30%** を目安にする

### 何を英語で残すか

| 英語で残す | 例 |
|-----------|-----|
| プロダクト名 | Claude Code, Codex, Docker, Kubernetes |
| 企業名 | Anthropic, Google, OpenAI, Block |
| 技術用語 | MCP, CI/CD, SaaS, LLM, API |
| 略語・頭字語 | COBOL, OAuth, JSON, YAML |
| コード・設定値 | `codex exec`, `--voice`, `keybindings.json` |
| 直接引用 | 原文のまま（日本語訳は地の文で補足） |

### 何を日本語にするか

| 日本語にする | 例 |
|-------------|-----|
| 文脈・背景説明 | 「2026年2月、太平洋を挟んで2つの大型人員削減が発表された」 |
| 分析・考察 | 「この構造的変化は日本のSI企業にも波及する」 |
| 感情・インパクト | 「衝撃」「急騰」「危機感」 |
| 日本固有の概念 | 「配置転換」「全銀システム」「メガバンク」 |

### 初出の技術用語

```
MCP（Model Context Protocol）は、AI エージェントが外部システムと通信するための標準規格。
```

- 英語略語 + 括弧内にフルネーム + 日本語での一言説明

## 10. リンクの扱い

### 参考リンク（末尾にまとめる）

```markdown
## 参考リンク

- [Anthropic Blog: Code Review](https://claude.com/blog/code-review)
- [TechCrunch: Anthropic launches code review tool](https://techcrunch.com/2026/03/09/...)
- [Bloomberg: Block to cut 40% of workforce](https://bloomberg.com/...)
```

### ルール

| ルール | 説明 |
|--------|------|
| **本文中にインラインリンクを散らさない** | 末尾の「参考リンク」にまとめる |
| **3〜10個** が目安 | 多すぎるとノイズになる |
| **ソースの質** | 一次ソース（公式ブログ、プレスリリース）を優先 |
| **アンカーテキスト** | 「こちら」「click here」は使わない。ソース名 + 記事タイトル |

## 11. 「日本への示唆」セクションの書き方

### このセクションが記事の付加価値

- 既存記事の **85%** に含まれる最重要セクション
- 海外ニュースをそのまま翻訳するだけでは価値がない
- 「日本ではどうなるか」「自分ごととしてどう捉えるか」を具体的に書く

### 書き方パターン

#### パターン1: 日本企業への波及

```markdown
## 日本への示唆

日本のメガバンクは COBOL 資産を大量に抱えている。
全銀システムの次期更改は 2027年に控えており、AI による自動マイグレーションが選択肢に入る。

| 銀行 | COBOL依存度 | リスク |
|------|-----------|--------|
| みずほ | 高（MINORI） | マイグレーション複雑性 |
| 三菱UFJ | 中 | 段階的移行中 |
```

#### パターン2: エンジニアのキャリアへの影響

```markdown
## 日本のエンジニアにとって

SaaS の「シート課金」モデルが崩壊すると、日本の SI 企業のライセンス再販ビジネスも影響を受ける。

「ツールを売る」から「成果を売る」への転換が求められる。
```

#### パターン3: 文化・制度の違いを対比

```markdown
## 日本との違い

| 観点 | 米国 | 日本 |
|------|------|------|
| 解雇速度 | 即時 | 配置転換（年単位） |
| 株価反応 | 好感（コスト削減） | 無反応〜微減 |
| 社会的受容 | 「効率化」として肯定 | 「リストラ」として批判 |
```

## 12. 所感・意見の入れ方

### バランス

- **事実 70% / 分析・意見 30%**
- 意見は分析から自然に導く。唐突な主張は避ける

### 所感セクション（任意）

- 技術リリース記事で「### 所感」として入れることがある
- 1〜3文で簡潔に
- 例: `Go で AI アプリを書くなら第一選択肢。型安全なのが Go らしい。`

### 意見を入れるときのルール

| OK | NG |
|----|-----|
| データに基づく分析 | 根拠のない推測 |
| 「この構造的変化は〜」 | 「たぶん〜だろう」 |
| 比較から導く結論 | 感情的な煽り |
| 「〜が加速する」（トレンドの延長） | 「〜に違いない」（断言しすぎ） |

---

## 13. 避けるべきこと

| NG | 理由 |
|----|------|
| 長い導入文 | すぐ本題へ。3文以内で核心に入る |
| 「〜と思います」「〜かもしれません」 | 断定調で書く |
| 全変更を網羅 | 重要なものを厳選。テーブルで整理 |
| 海外記事のそのまま翻訳 | 自分の言葉で解釈し、日本への示唆を加える |
| インラインリンクの乱用 | 末尾にまとめる |
| 絵文字の多用 | テーブル内のステータス表示（✅, 🔬）程度に留める |
| 「こちらの記事によると」 | ソースは参考リンクで示す。本文は地の文で書く |
| 「〜の違い:」「〜の特徴:」の後に箇条書き | AI生成感が出る。テーブルか地の文で書く |
| 1段落が5文以上 | 分割する |

## 14. 学習中のトピックとの接続

ai/ リポジトリで学習中の内容と最新ニュースを掛け合わせてブログネタを作る。

### 現在学習中のトピック（優先的にネタにする）

| トピック | 関連知識 |
|---|---|
| コンテキストエンジニアリング | Transformer, ロール指定, Function Calling, RAG, Reasoning |
| LLM APIサービス | OpenAI Responses API, Conversations API, Vertex AI, Bedrock |
| エージェント設計 | Agent Teams, MCP, Sub-agents, Tool Use |
| ファインチューニング | LoRA, SFT, 小型モデル, MLX |
| LLM Observability | トレーシング, トークン統計, ハルシネーション検出 |
| セキュリティ | APIキー管理, サプライチェーン攻撃, Jailbreak防御 |
| マネジメント × AI | エンパワメント型, 1on1, バイブコーディング, AI PM |

### ネタの掛け合わせ方

1. **昨日〜今日の**最新ニュースを検索（鮮度が最優先）
2. 学習中のトピックとの接点を探す
3. 書籍で学んだ理論 + ニュースの事実 + 自分の実装経験 を組み合わせる

例:
- 「OpenAI Conversations API リリース」× コンテキストエンジニアリング Ch2 の知識
- 「LiteLLM サプライチェーン攻撃」× セキュリティ実体験
- 「Gemini APIキー脆弱性」× Browser key 不正利用インシデント

### ルール

- 書籍の内容をそのまま転載しない（著作権配慮）
- 最新ニュースをフックにして、学んだ知識で深掘りする構成
- 自分の実装経験（mcp/, agent/, ft/）があれば具体例として含める
