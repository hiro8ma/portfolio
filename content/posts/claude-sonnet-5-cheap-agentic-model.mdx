## 概要

Anthropic は2026年6月30日、Claude Sonnet 5 を発表した。

Sonnet 5 は同社の中位モデルの最新版である。
7月1日から、Claude Code と free / Pro プランの既定モデルになった。

発表の力点は性能ではなく価格対効果にある。
Sonnet 5 は Opus 4.8 に迫る agentic 性能を、その数分の一の価格で提供する。
導入価格は入力$2、出力$10（100万トークンあたり）で、8月31日まで適用される。

この発表が示すのは、競争軸の移動である。
勝負どころが「最も賢いモデル」から「十分な性能を安く大量に回せるモデル」へ動いた。

## 詳細

### 何が出たのか

Sonnet 5 は、Anthropic が「これまでで最も agentic な Sonnet」と位置づけるモデルである。

計画を立て、browser や terminal などのツールを使い、自律的に多段のタスクを最後まで実行する。
数か月前なら、この水準の自律動作にはより大きく高価なモデルが必要だった。
native の 1M トークンのコンテキストウィンドウを備える。

提供範囲は広い。
free と Pro の既定モデルであり、Max、Team、Enterprise でも使える。
API では `claude-sonnet-5` で呼び出す。

### Opus 4.8 にどこまで迫るか

性能は、上位の Opus 4.8 に近い水準まで来た。
ただし追い抜いてはいない。

agentic coding のベンチマークでは、Sonnet 5 が Opus 4.8 に届かない。
一方で knowledge work では、Sonnet 5 が Opus 4.8 をわずかに上回る。
Anthropic 自身、高い精度が要る作業では Opus 4.8 が引き続き第一選択だと明言している。

| ベンチマーク | Sonnet 5 | Sonnet 4.6 | Opus 4.8 |
|-------------|---------|-----------|---------|
| agentic coding | 63.2% | 58.1% | 69.2% |
| knowledge work | Opus 4.8 をわずかに上回る | 下回る | 基準 |

Sonnet 5 の狙いは、Opus 4.8 を倒すことではない。
Opus に迫る性能を、桁の違う価格で届けることにある。

### 価格がエージェント運用を左右する

価格を見ると、Sonnet 5 の狙いが明確になる。

導入価格は入力$2、出力$10で、8月31日まで続く。
その後は入力$3、出力$15へ上がる。
それでも Opus 4.8、GPT-5.5、Gemini 3.1 Pro より安い。

| 区分 | 入力（$/Mtok） | 出力（$/Mtok） | 期間 |
|------|--------------|--------------|------|
| 導入価格 | $2 | $10 | 8月31日まで |
| 標準価格 | $3 | $15 | 9月1日以降 |

価格が効く理由は、エージェントのトークン消費の構造にある。
エージェントは、計画、ツール呼び出し、検証を繰り返すたびにトークンを消費する。
ループで動かす場合、周回ごとにコンテキストを再送するため、消費量は周回数に対して積み上がる。

ここで単価が効く。
同じ作業を回すとき、トークン単価が半分なら、運用コストも半分に近づく。
高精度が不要な反復作業では、Opus 級のモデルを使う必要はない。
十分な性能の中位モデルを安く回すほうが、採算に乗る。

> 「以前は途中で止まっていた。日々の自動化には、もう迷う理由がない」
> — Zapier シニアエンジニア（TechCrunch 経由）

## 日本への示唆

この価格対効果の改善は、日本企業の AI 導入判断を変える。

第一に、AI を載せられる業務の範囲が広がる。
推論の単価が下がると、これまで採算が合わなかった定型業務にも AI を回せる。
賢さの上限ではなく、単価あたりの実用性が導入範囲を決める変数になる。

第二に、モデルの使い分けが運用設計の中心になる。
高精度が要る工程には Opus 4.8 を、反復する自動化には Sonnet 5 を割り当てる。
この振り分けを設計できるかどうかで、同じ成果でも運用コストが変わる。

| 用途 | 推奨モデルの型 | 理由 |
|------|--------------|------|
| 高精度が要る判断・生成 | Opus 級 | 精度が成果を左右する |
| 反復する自動化・エージェント | Sonnet 5 級 | 単価が運用コストを左右する |
| 大量の下処理・分類 | さらに安価な小型モデル | 量が支配的 |

第三に、SaaS の従量課金モデルへの影響がある。
エージェントの運用コストが下がると、成果あたりの課金は競争にさらされる。
日本の SI 事業者やツール提供者は、モデルの単価低下を前提に価格設計を組み直す必要がある。

日本のエンジニアが読み取るべき点は、モデル選定が「どれが最強か」から「どの工程にどの単価を当てるか」へ移ったことである。
エージェントを設計するとき、精度と単価の割り当てが、そのまま事業の採算を決める。
